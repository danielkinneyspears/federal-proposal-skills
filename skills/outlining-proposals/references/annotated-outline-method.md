# Annotated Outline Method

How to build a proposal's structure from Section L, map requirements to
sections, annotate each section, and budget pages.

## Outline vs. annotated outline

A plain outline is a list of headings. An annotated outline adds, to each
heading, everything the writer needs to draft that section well: the
requirements it must satisfy, the factors it is scored under, the theme it
carries, the proof it lands, the graphics it uses, its page budget, and its
owner. The annotation is the value. A writer handed an annotated outline section
knows exactly what success looks like before writing a word.

## Building the structure from Section L

Section L is the authoritative source for proposal structure. Build the outline
to it:

1. Create the volume structure Section L defines (Technical, Management, Past
   Performance, Cost/Price, and any others).
2. Within each volume, create the sections in the order and with the headings
   Section L prescribes. If Section L gives explicit headings, use them verbatim.
3. Mirror Section L's numbering where it provides one. An evaluator scoring
   against Section L should be able to follow the proposal section by section
   without searching.

The evaluator's ease of scoring is a real evaluation advantage. A proposal that
forces an evaluator to hunt for where a requirement was answered loses credit it
earned.

## When Section L is thin

Task-order and commercial solicitations sometimes give minimal instructions.
When Section L does not prescribe sub-structure:

- Structure to **Section M** — make the sections follow the evaluation factors
  and subfactors, so each factor has an obvious home.
- Within a factor's section, structure to the **Section C/SOW/PWS** requirements
  it covers.
- Keep it conventional. An evaluator should never be surprised by the structure.

## Mapping requirements to sections

Every requirement in the compliance matrix is assigned a single home section:

- Work through the matrix row by row. For each requirement, decide the one
  section where the proposal will address it, and record it in the matrix's
  "Volume & section" column.
- A requirement that seems to belong in two places belongs primarily in one;
  the other section may reference it, but it is drafted and verified once.
- After mapping, scan for orphans (requirements with no section) and for
  sections that logically own requirements not yet assigned to them.

## Annotating a section

For each section in the outline, record:

- **Requirements covered** — the compliance-matrix requirement IDs.
- **Section M factor(s)** — what this section is scored under.
- **Win theme** — the one theme this section threads (from `05-win-strategy.md`).
- **Proof points** — the specific evidence the section must land.
- **Graphic concept(s)** — see `storyboarding.md`.
- **Page allocation** — from the page budget.
- **Author** — the owner responsible for drafting.

A section with no theme and no proof points is a compliance-only section. Some
sections genuinely are (administrative content); flag the ones that should carry
strategy but currently do not.

## Page budgeting

The page limit is fixed; the budget allocates it deliberately:

1. Start from the volume's page limit (from the format register).
2. Reserve pages for required front matter, graphics, and any mandated content.
3. Allocate the remainder across sections **weighted by Section M importance**.
   The factor that most decides the award gets the most pages. A factor weighted
   lightly in Section M does not earn equal pages just because it has many
   requirements.
4. Confirm the total is at or under the limit. If it overflows, cut from the
   lowest-value sections, not proportionally.

The page budget is a constraint the writers work within, and a planning signal:
if a high-value section cannot fit its content in its budget, that is a
structural problem to solve now, not at the Red Team.
